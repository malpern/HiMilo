VoxClaw

Hacker Dojo · Talk

Giving Your AI Agent a Voice.

Your agent runs on a Mac Mini under the desk. It finishes a task and… nothing. No sound, no signal. VoxClaw is a menu bar app that listens on the network, speaks with neural voices, and shows a teleprompter overlay — so your agent can talk to you.

One HTTP endpoint. One curl command. Thirty seconds from silent agent to speaking assistant. This is the story of building it — from a simple CLI to a multi-agent voice system that knows when to shut up.

Start the story →

VoxClaw started as a weekend hack: can I make my agent say something out loud? Thirty lines of Swift, Apple TTS, done. But then the questions piled up. What if the agent is on another machine? What if two agents are talking? What if I'm on a Zoom call? What if I just want it to know I'm listening?

Each question became a feature. Each feature taught something about the surprisingly tricky space between text and speech. This talk walks through seven chapters of that evolution, ending with where it's going next: an AI camera that reads your gaze and takes turns like a human would.

The landing page is at malpern.github.io/VoxClaw — the source is on GitHub.

Chapter I

The silent agent problem

"The best interface is no interface." Except when your agent finishes a ten-minute task and you don't notice for an hour.Every agent developer, eventually

Picture the setup: you have an AI agent — maybe it's Claude Code, maybe it's a custom pipeline — running on a Mac Mini, a server, or just a different terminal tab. It fetches data, writes code, runs tests. It finishes. And then… it updates a log file. Maybe it sends a notification that gets buried under thirty others.

The fundamental mismatch is this: agents are getting better at doing work, but they're terrible at announcing work. A notification badge is not a conversation. A log line is not a report. What you actually want is for your agent to say something — out loud, in a human voice, at the right moment.

But "just add TTS" runs into problems immediately:

  • The agent might be on a different machine than your speakers
  • You might be on a Zoom call — the agent shouldn't interrupt
  • Three agents might finish at the same time — who talks first?
  • You want to see what it's saying, not just hear it
  • Different agents should sound different so you know who's talking

These aren't edge cases. They're the first five things that happen when you actually try to give an agent a voice. VoxClaw is the result of solving them one at a time.

The premise

Speech is the missing output channel for AI agents. Not chat, not notifications — actual voice. But doing it well means solving network delivery, interruption awareness, multi-agent coordination, and visual feedback.

Chapter II

One endpoint, thirty seconds

The best API is one line of curl.The VoxClaw design principle

VoxClaw runs as a macOS menu bar app. When it starts, it opens an HTTP listener on port 4140 and advertises itself on the local network via Bonjour. Any agent on the same LAN can find it and send it text to speak.

The entire API surface for basic use is one endpoint:

curl -X POST http://your-mac.local:4140/read \
  -H 'Content-Type: application/json' \
  -d '{"text": "Build complete. All 47 tests passed.", "voice": "onyx"}'

That's it. The Mac speaks the text, shows a floating overlay with word-by-word highlighting, and dismisses the panel when it's done. No SDK, no WebSocket, no auth token for local use.

Three endpoints, three jobs

  • POST /read — Speak text immediately or queue it. Accepts text, voice, rate, and instructions for speaking style.
  • POST /agent-notify — Structured notifications with kind (summary, progress, failure) for different treatment.
  • GET /status — Health check that returns the current state: idle, loading, playing, paused, finished. Useful for agents that want to wait their turn.

Discovery via Bonjour

The agent doesn't need to know your Mac's IP address. VoxClaw advertises a _voxclaw._tcp service on the local network. Any agent with a Bonjour client can discover it automatically — zero configuration, zero hardcoded addresses.

Why HTTP, not WebSocket?

WebSocket would let VoxClaw push state changes back to the agent. But agents are fire-and-forget speakers — they rarely need to know when speech finishes. HTTP keeps the integration dead simple: every language has curl or an HTTP client. The browser extension polls /status on a timer, which is good enough for a UI update every second.

Thirty-second setup

Install VoxClaw, pick a voice, done. The onboarding wizard generates a copy-paste config block your agent can consume — URLs, port, voice preference, health check endpoint.

Chapter III

Live demo

Show, don't tell. Especially when the thing you're showing literally talks.

This chapter is a live demo during the talk. Here's the script:

Demo 1 — Basic speech

Open a terminal. Curl a message to VoxClaw. Watch the overlay appear with word-by-word highlighting. Hear the neural voice.

# Basic speech
curl -X POST http://macbook-air-m4.local:4140/read \
  -d '{"text": "Hello Hacker Dojo. This is VoxClaw speaking."}'

# Check status
curl -s http://macbook-air-m4.local:4140/status | python3 -m json.tool
Demo 2 — Different voices

Send two messages with different voices back-to-back. Show the queue in action — the second waits for the first to finish, then speaks in a different voice.

# Agent 1 speaks with "onyx"
curl -X POST http://macbook-air-m4.local:4140/read \
  -d '{"text": "Deploy finished. All services healthy.", "voice": "onyx"}'

# Agent 2 speaks with "nova" — queued behind the first
curl -X POST http://macbook-air-m4.local:4140/read \
  -d '{"text": "I found three unused API keys in the config.", "voice": "nova"}'
Demo 3 — Keyboard controls

Send a longer text. Press Space to pause/resume. Press arrow keys to skip. Press Escape to stop.

What you just saw

Network speech delivery, word-synchronized overlay, voice selection, queuing, and keyboard control — all from curl.

Chapter IV

How it evolved

Every feature started as a problem that showed up on the second day of using the previous feature.

VoxClaw didn't start as a multi-agent voice system. It started as thirty lines of Swift that called Apple's built-in TTS. Here's the path from there to here, in roughly chronological order.

1. A simple CLI

The first version was voxclaw "Hello world" — a command-line tool that spoke text using Apple's AVSpeechSynthesizer. Fine for a demo. Useless for an agent on another machine.

2. Network listener

The first real feature: an HTTP server on port 4140. Now an agent running anywhere on the LAN could POST text and hear it come out of the Mac's speakers. This was the "aha" moment — speech as a network service.

3. Bonjour discovery

Hardcoding IP addresses is fragile. VoxClaw advertises _voxclaw._tcp via Bonjour, so agents find it automatically. Zero config. The agent doesn't even need to know which machine has the speakers.

4. Teleprompter overlay

Hearing speech is good. Seeing it is better. A floating NSPanel appears over everything, highlighting each word as it's spoken. This turned out to be the hardest feature — synchronizing text highlighting to audio playback at the word level (more on this in chapter V).

5. Multi-agent support with distinct voices

Once you have two agents reporting to VoxClaw, you need to tell them apart. Each agent can specify a voice — onyx, nova, echo, shimmer — and VoxClaw remembers per-project voice assignments. You hear "onyx" and know it's the deploy agent; you hear "nova" and know it's the code reviewer.

6. Speech queue and agent turn-taking

Two agents finishing at the same time means two speech requests arriving simultaneously. VoxClaw maintains a FIFO queue with a 20-item cap. Agents take turns. Smooth fade transitions between speakers give each message a natural boundary. A one-second pause between items provides a "breath" — a subtle cue that a new speaker is about to talk.

7. Context awareness — knowing when to shut up

This was the feature nobody asked for until it was obvious. VoxClaw enumerates running audio processes via CoreAudio. If it detects a Zoom call, a Teams meeting, or FaceTime, it auto-pauses. When the call ends, it resumes. If you start talking, it yields. It checks every second with a 150-second timeout fallback.

8. iOS companion app

Shared VoxClawCore library, iCloud KVS for settings sync. Your iPhone can be a relay target — speech arrives at the Mac and forwards to the phone via peer-to-peer. Cross-device pause/resume via a /control endpoint.

9. Browser extension

A Chrome extension that polls /status and gives a visual indicator of what VoxClaw is doing. No WebSocket needed — a one-second poll loop is perfectly fine for a status badge.

10. App Store dual-distribution

The "polite-wait" CoreAudio feature doesn't work in the App Store sandbox. Solution: ship two builds. The direct download has full functionality; the App Store build compiles out the sandboxed features and ships with a feature-gated subset. Same codebase, different entitlements.

Ten layers deep

Each layer solved the problem the previous layer created. CLI → network → discovery → overlay → multi-voice → queue → context → cross-device → browser → App Store. The common thread: speech is a systems problem, not a TTS problem.

Chapter V

Three hard problems

The easy part is making the computer talk. The hard part is making it talk at the right time, in the right way, and in sync with what's on screen.

Behind VoxClaw's simple API are three problems that took disproportionate effort to solve. Each one looks trivial from the outside.

Problem 1: Audio-text synchronization

When VoxClaw speaks a paragraph, the overlay highlights each word in real time. This requires knowing exactly when each word is being spoken. The problem: not all TTS engines give you that information.

VoxClaw uses three progressively better strategies:

1 Cadence-based Estimate duration from word count, distribute evenly
↓
2 Proportional Weight by character count — "the" gets less time than "synchronization"
↓
3 Aligned OpenAI returns precise per-word timestamps — pixel-perfect sync

The system starts with cadence-based timing (instant, available before audio loads) and upgrades transparently when better data arrives. The user never sees a loading state or a misaligned highlight — the worst case is slightly imprecise timing, not broken timing.

Problem 2: Polite-wait via CoreAudio

Detecting whether you're on a video call sounds like it should be a one-liner. It is not.

macOS doesn't have an "is the user on a call?" API. What it does have is kAudioHardwarePropertyProcessObjectList — a way to enumerate every process using the audio hardware, and whether each one is actively running input (microphone) or output (speakers).

VoxClaw polls this every second. When it sees Zoom, Teams, FaceTime, or any app actively using both input and output, it pauses the speech queue. When the call ends (the process stops using audio), it resumes. A 150-second timeout prevents getting stuck forever if the detection goes stale.

The sandbox problem

This CoreAudio enumeration requires entitlements that the App Store sandbox doesn't allow. VoxClaw solves this with conditional compilation: the direct-download build includes polite-wait; the App Store build compiles it out entirely and falls back to manual pause. Same source, two targets, no runtime feature flags.

Problem 3: Queue coordination

The speech queue coordinator is the most regression-prone component in VoxClaw. It seems simple — a FIFO queue with a play head — but the edge cases multiply:

  • An agent sends 15 items in a burst. The queue cap is 20. What happens at 21?
  • The user pauses mid-sentence. A new item arrives. Does it queue behind the paused item, or interrupt it?
  • The TTS engine hangs (network timeout, API error). The queue is stuck. How long do you wait before declaring the current item dead?
  • Two items from different agents arrive 200ms apart. The fade-out from item 1 overlaps with the fade-in of item 2. Do they cross-fade, or does one wait?

The answers: drop at 21 with an error response. Queue behind, never interrupt. Five-minute timeout kills zombie streams. Sequential — fade-out completes, one-second gap, then fade-in. Each answer was earned from a bug report.

The takeaway

The API is one line of curl. The implementation behind that line handles three timing systems, a CoreAudio process enumerator, and a queue coordinator with five-minute deadman switches. Simple APIs require complex internals.

Chapter VI

Architecture

Protocols, not inheritance. Modules, not folders.

VoxClaw is a SwiftPM package with two targets: VoxClawCore (the library, 56 Swift files) and VoxClaw (a thin executable that imports the core). The iOS companion app imports the same core. This means every feature is implemented once and tested once.

Thirteen modules

Audio Speech engines, playback, timing
Network HTTP listener, Bonjour, peer browsing
Reading Speech sessions, word timing
Input CLI args, stdin, file, clipboard, URL scheme
Panel Floating overlay (NSPanel + SwiftUI)
Coordination Speech queue, session management
Voice Voice pool, per-project bindings
Settings Preferences, iCloud KVS sync
Views Menu bar, settings, onboarding

The speech engine protocol

The key architectural decision: speech engines are protocol-based. Apple, OpenAI, and ElevenLabs all conform to the same interface. The coordinator doesn't know or care which engine is active. Adding a new TTS provider means implementing one protocol — no changes to the queue, the overlay, or the network layer.

Fallback is automatic: if the OpenAI API key is invalid or the network is down, VoxClaw falls back to Apple's built-in TTS. The user hears a different voice but never sees an error dialog. A SilentEngine exists for cases where even Apple TTS shouldn't fire (iOS system mute, testing).

By the numbers

  • 56 Swift source files across 13 modules
  • 75+ unit tests
  • Swift 6.2 with strict concurrency
  • Targets macOS 26+ and iOS 26+
  • Three speech engine implementations + two test engines
  • Five input methods (CLI, stdin, file, clipboard, network)

Architecture principle

One library, two platforms, three engines, five inputs. The protocol-based engine abstraction is the load-bearing decision — everything else follows from it.

Chapter VII

Future: natural turn-taking

The best wake word is no wake word. Just look at me.

Everything VoxClaw does today is one-directional: the agent speaks, you listen. The next step is making it a conversation — but not with a wake word, not with a push-to-talk button. With eye contact.

The AI camera

The idea: a camera component (built-in Mac camera or an external one) that uses vision AI to detect whether you're looking at the device. When you look at it, transcription activates — VoxClaw starts listening. When you look away, it stops. No "Hey Siri," no button press. The same signal humans use to manage turn-taking in real life.

Visual eyes

But gaze detection alone isn't enough. You need feedback. When VoxClaw detects you're looking at it, it should acknowledge that — visually, immediately, unmistakably.

The plan: animated eyes on the overlay that respond in real time:

  • You look at it — the eyes make eye contact back. Pupils track your position. A subtle widening says I see you.
  • It's speaking — the eyes animate expressively. Blinks, shifts, slight widening on emphasis. Not a static icon — a presence.
  • You look away — the eyes look away too, or close. Mutual disengagement. The transcription stops. The agent knows you're done.
  • It's listening — the eyes are attentive, steady, slightly wider. A visual signal that it's your turn to speak.
The goal isn't to build a face. It's to build a social signal. The minimum viable presence that makes an AI agent feel like it's in the room with you — not just reading text into a speaker.

Why this matters

Every voice assistant today uses an explicit trigger: a wake word, a button, a keyboard shortcut. These work, but they break the conversational metaphor. You don't say "Hey Human" before talking to a colleague. You make eye contact, and they know it's their turn.

If VoxClaw can detect gaze reliably and respond with visual acknowledgment, it collapses the distance between "AI tool" and "AI presence." Not a chatbot. Not a smart speaker. Something closer to a colleague sitting across the desk who happens to be an AI.

The vision

Replace wake words and push-to-talk with the same social cues humans already use — eye contact, gaze aversion, turn-taking. Make the agent feel present, not just audible. Give it eyes that see you seeing it.
